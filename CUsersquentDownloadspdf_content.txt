=== PAGE 8 ===
SQL vs NoSQL : quand choisir ?
SQL (PostgreSQL):
• Schéma fixe et bien défini
• Relations entre entités (clés étrangères)
• Requêtes complexes (JOIN, agrégations)
• Contraintes d'intégrité garanties
NoSQL (MongoDB):
• Schéma flexible (documents JSON)
• Structure variable selon les enregistrements
• Pas de jointures complexes nécessaires
• Évolution facile du modèle

=== PAGE 11 ===
Docker : Volumes & Réseaux
Deux concepts nouveaux pour ce projet :
Volumes:
• Persistent les données même si le conteneur est supprimé
• Sans volume, toutes les données PostgreSQL sont perdues au docker compose down
• Déclaré sous volumes: dans le docker-compose
Réseaux:
• Réseau isolé pour que les conteneurs communiquent entre eux
• Un service se connecte à un autre via son nom de service (ex: postgres, pas localhost)
• Déclaré sous networks: dans le docker-compose

=== PAGE 15 ===
Structure d'un docker-compose.yml
Clé Rôle
services: Déclare chaque conteneur
image: Image Docker à utiliser (ex: postgres:15)
environment: Variables passées au conteneur (via .env)
volumes: Monte un volume pour persister les données
networks: Relie les services dans un réseau isolé
depends_on: Ordre de démarrage (avec condition: service_healthy)
healthcheck: Vérifie que le service est vraiment prêt
ports: Expose un port sur l'hôte (hôte:conteneur)

=== PAGE 16 ===
Variables d'environnement
Règle : jamais de mot de passe en dur dans le code ou le docker-compose.yml.
Fichier .env (à la racine, jamais commité):
Dans docker-compose.yml, on référence avec ${VAR}:

=== PAGE 17 ===
Healthcheck & depends_on
Problème :
Un conteneur "démarré" n'est pas forcément "prêt"
à recevoir des connexions.
Sans healthcheck, l'ingestor peut tenter de se connecter
avant que la base soit prête: risque d'erreur au démarrage.
